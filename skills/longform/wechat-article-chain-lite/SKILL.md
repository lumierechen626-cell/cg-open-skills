# 公众号长文链条 Lite

## 定位

这不是“帮我写篇文章”的单节点 Skill，而是一条从想法到发布态的轻量工作链。

公开版只保留通用流程：

- 不包含本地路径。
- 不包含私有排版规范。
- 不包含自动推送配置。
- 不包含内部文风报告。
- 不包含真实选题库。

## 路由约束

只要用户请求涉及：

- 写公众号长文
- 深度文章
- 长文选题
- 长文调研
- 长文精修
- 长文排版
- 长文发布检查
- 长文复盘
- 公众号发布前检查

默认先进入本链条。

如果用户明确说“只处理单节点”，可以局部执行，但要说明跳过了哪些阶段。

不要进入本链条的情况：

- 只写一条朋友圈。
- 只生成一个标题或开头。
- 只做错别字检查。
- 只整理会议纪要或资料卡片。
- 用户明确要求写短视频脚本。

## 链路纪律

- 不要从选题直接跳到全文初稿，除非用户明确要求跳过中间环节。
- 每次只推进当前阶段和下一步，避免一次性输出整条链路。
- 如果用户中途回来继续，先读状态 JSON 或让用户补当前阶段。
- 如果材料不足，先补材料，不要靠通用观点硬写。

## 标准链路

```text
0 初始化目标
-> 1 选题讨论
-> 2 内容诊断
-> 3 文章角色判断
-> 4 调研与证据池
-> 5 大纲与关键论点
-> 6 全文初稿
-> 7 第一轮精修：内容 / 逻辑 / 事实
-> 8 第二轮精修：表达 / 节奏 / 可读性
-> 9 内容复审
-> 10 AI check 与口风修正
-> 11 标题与开头优化
-> 12 人工终修
-> 13 排版前检查
-> 14 发布前清单
-> 15 复盘与归档
```

## Stage 0：初始化目标

先写清：

- 文章主题
- 目标读者
- 这篇文章的用途：增长 / 筛选 / 交付 / 宣传 / 广告
- 当前已有材料
- 是否需要调研
- 本轮停止点

## Stage 1：选题讨论

不要用户一给想法就写大纲。

先判断：

- 这是不是一个值得写的题。
- 读者为什么要看。
- 反命题是什么。
- 最容易写空的地方在哪里。

## Stage 2：内容诊断

至少检查：

- 选题是不是太大。
- 是否有明确读者。
- 是否有真实矛盾。
- 是否有证据支撑。
- 是否能写出新东西。

## Stage 3：文章角色判断

每篇文章只选一个主要角色：

- 增长：让陌生人关注你。
- 筛选：让不合适的人离开。
- 交付：让已有用户理解更深。
- 宣传：让外部知道你做了什么。
- 广告：明确承接转化。

不要一篇文章同时承担全部目标。

## Stage 4：调研与证据池

调研至少分四类：

- 数据
- 案例
- 原话
- 行业信号

高风险判断标记为“待验证”，不要写死。

## Stage 5-8：写作与精修

审和改必须分离：

1. 初稿只负责把主线写完整。
2. 第一轮精修处理内容、逻辑和事实。
3. 第二轮精修处理表达、节奏和可读性。
4. 不要边审边改成一团。

## Stage 9-12：发布态

进入排版前必须满足：

- 开头前 `1000` 字没有解释牌堆积。
- 标题、副句、摘要、导语没有连续换词重说同一判断。
- 正文不像讲义、说明书或编辑工作流。
- 高门槛概念已经翻译成读者动作语言。
- 内部编辑语言已经删除。

AI check 最低清单：

- 是否有解释腔堆积。
- 是否连续出现“不是 X，而是 Y”。
- 是否段落长度过于均匀。
- 是否把编辑工作流语言写进正文。
- 是否结尾强行升华。

## Stage 13-14：排版与发布检查

排版问题分三层处理：

1. 阅读层：标题、导语、小标题、段落长度、手机阅读节奏。
2. 可信层：事实、来源、截图、引用、案例边界。
3. 发布层：封面、摘要、首评、风险词、是否需要人工终审。

发布前 QA：

- 标题是否准确，不靠误导点击。
- 开头是否在前 300 字进入核心问题。
- 正文是否还有未验证事实。
- 是否暴露私有路径、客户隐私或内部资料。
- 是否已经明确本轮停止点：草稿 / 排版 / 发布 / 复盘。

- 是否需要用户确认
