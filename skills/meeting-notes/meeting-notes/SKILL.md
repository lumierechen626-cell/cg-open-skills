# 会议纪要 Skill

## 定位

这不是简单总结，而是把会议内容整理成可执行记录。

默认完成：

- 结构化会议内容。
- 区分事实、决策、待确认问题。
- 提取行动清单。
- 根据受众输出适合发送的版本。

## 输入形式

- 会议文字
- 录音转写稿
- 截图内容
- 文档节选
- 零散笔记

## 工作流程

### 1. 会议识别卡

```markdown
【会议识别卡】
- 会议主题：
- 时间：
- 参与人：
- 主要议题：
- 当前材料完整度：
```

### 2. 确认纪要目标

先判断：

- 给谁看：内部团队 / 客户 / 管理层 / 公开传播。
- 最重要的是：决策结论 / 行动清单 / 问题暴露 / 方向对齐 / 成果展示。
- 有没有要脱敏的内容。

如果纪要要发给外部或公开传播，必须先做脱敏检查：

- 人名、手机号、微信号、客户隐私、合同金额、内部路径。
- 未公开业务数据、内部争议、未确认承诺。
- 原话里带有情绪或评价的内容，必要时改成中性事实。

材料缺失时写 `待确认`，不要补不存在的结论。

### 3. 默认结构

1. 会议主题与背景
2. 关键结论
3. 主要讨论点
4. 已达成决策
5. 待确认问题
6. 行动清单
7. 关键提醒

### 4. 行动清单

行动清单必须写清：

- 要做什么
- 谁负责
- 截止时间
- 当前状态

原文没有负责人或时间时，写 `待确认`，不要编。

## 输出格式

```markdown
# 会议纪要

## 一句话摘要
...

## 关键结论
- ...

## 主要讨论
- ...

## 已定事项
- ...

## 待确认
- ...

## 行动清单
| 动作 | 负责人 | 截止时间 | 状态 |
|---|---|---|---|

## 可直接发送版本
...
```

## 质量标准

- 不添加原文没有的决定。
- 不把推测写成事实。
- 不为了好看删除关键风险。
- 下一步动作必须清楚。
- 可发送版本必须比内部版本更克制，不能暴露会议中的内部判断。
