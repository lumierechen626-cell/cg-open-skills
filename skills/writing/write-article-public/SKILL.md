# 长文写作

## 定位

这只 Skill 处理单篇文章写作，不负责完整公众号发布链路。

如果用户要从选题到发布全流程，应使用长文链条。

## 工作流程

### 1. 确认文章任务

先确认：

- 写给谁看。
- 文章想解决什么问题。
- 文章类型：观点、教程、复盘、行业分析、案例拆解。
- 已有材料有哪些。
- 是否需要调研。

先输出材料盘点卡：

```markdown
【材料盘点卡】
- 已有观点：
- 已有事实：
- 已有案例：
- 已有数据：
- 缺口：
- 是否必须调研：
```

### 2. 建主线

先写：

- 核心判断
- 反命题
- 读者为什么要关心
- 需要证明什么

主线卡：

```markdown
【文章主线卡】
- 核心判断：
- 反命题：
- 这篇文章反对什么：
- 读者看完应改变什么判断：
- 最需要证明的一句话：
```

必须调研的情况：

- 涉及最新政策、行业数据、平台规则或市场变化。
- 涉及具体公司、产品、价格、法律、医疗、金融事实。
- 文章要公开发布并带有明确判断。
- 用户提供的材料只有观点，没有证据。

### 3. 写大纲

大纲必须说明每一节的作用，不只列标题。

### 4. 写初稿

初稿先求完整，不追求漂亮。

### 5. 精修

分两轮：

- 内容轮：逻辑、事实、证据。
- 表达轮：节奏、可读性、开头、结尾。

发布前检查：

- 事实：关键数据、时间、人物、机构和引用是否可回溯。
- 逻辑：每一节是否在推进主线，而不是重复观点。
- 风险：是否有过度承诺、泄露隐私、未经核查的判断。
- 表达：是否像讲义、工作流或 AI 总结。

## 输出结构

```markdown
# 文章写作包

## 文章任务
- 目标读者：
- 核心判断：
- 文章类型：

## 大纲
1. ...
2. ...

## 初稿
...

## 精修建议
- 内容问题：
- 表达问题：
- 需要补的证据：
```

## 规则

- 不假设用户有内部知识库。
- 没有证据的判断要标出来。
- 不把摘要写成压缩版正文。
- 不把长文写成讲义或说明书。
- 没有证据时可以先写观点草稿，但必须标注“待补证据”。
